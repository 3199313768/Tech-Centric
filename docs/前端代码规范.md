# Part A 中级基线

行业通用最佳实践。未特别说明时，默认按此执行。

## A1. 总则

### 最小改动

- 只改任务相关文件与代码路径
- 不做无关重构、不批量改格式、不「顺手」升级依赖
- 抽象在**第二处**重复时再提取

### 类型安全

- `strict: true` 下禁止 `any`
- 禁止无说明的 `@ts-ignore`；必须附原因且收窄范围
- 外部输入先校验再使用

### 可读性

- 命名自解释；布尔值用 `is/has/can` 前缀
- 注释只写非 obvious 业务规则
- 单文件建议 ≤ 250 行，超出则拆分

### 可访问性

- 交互控件有 `aria-label`、可见文本或 `<label htmlFor>`
- 图标按钮必须 `aria-label`
- 模态框：`role="dialog"`、`aria-modal`、焦点管理

## A2. 目录与导入

```
src/
├── app/              → 路由、layout、metadata、Route Handler
├── components/       → UI（按功能域分子目录）
│   ├── home/         → 首页 Tab 区块与全局壳层
│   ├── knowledge/    → 知识库
│   ├── rag/          → AI 助手（仅 UI）
│   └── spirit/       → 设计系统复用组件
├── lib/              → 基础设施与业务逻辑，不含 JSX
│   ├── knowledge/    → types.ts、queries.ts
│   ├── rag/          → RAG + contactFlow
│   └── supabase/
├── data/             → 静态种子数据
└── utils/            → 纯函数 + hooks
```

- 导入统一 `@/`，禁止 `../../../` 超过两层
- 页面逻辑过重 → 拆到 `lib/<domain>/` 或 Server Actions

## A3. TypeScript

| 类型 | 约定 | 示例 |
|------|------|------|
| Props | `ComponentNameProps` | `RecordCardProps` |
| 领域实体 | `PascalCase` | `KbRecord` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_FILE_SIZE` |
| 函数/变量 | `camelCase` | `fetchRecords` |

- 复用类型：`src/lib/<domain>/types.ts`
- 异步显式返回类型；`catch (e: unknown)` + 类型收窄

```typescript
// ✅ GOOD
catch (e: unknown) {
  const message = e instanceof Error ? e.message : '操作失败'
  setError(message)
}

// ❌ BAD
catch (e: any) { alert(e.message) }
```

## A4. React / Next.js

| 场景 | 做法 |
|------|------|
| 默认 | Server Component |
| hooks / 事件 / 浏览器 API | 最小 Client 子组件 |
| 数据读取 | Server Component 内 `await` |
| mutation | Server Actions 或 Route Handler |
| searchParams | `Promise<...>` + `await` |

- 页面 / layout：`export default`
- 可复用组件：**named export**
- 禁止无 UI 的 `.limit(N)` 截断
- `useEffect` 仅浏览器侧副作用，需 cleanup
- 每路由提供 `metadata` 或 `generateMetadata`

```tsx
// ❌ BAD — 整页 client 拉数
'use client'
useEffect(() => { fetch('/api/x') }, [])

// ✅ GOOD
export default async function Page({ searchParams }: PageProps) {
  const params = await searchParams
  const supabase = await createClient()
  // await query...
}
```

## A5. 错误处理

- **禁止** `alert()` / `confirm()` 作为用户反馈
- **禁止** 仅 `console.error` 而无 UI
- **禁止** 吞 error 仍渲染「空成功」
- 表单：内联 error；列表：fallback；提交中：`disabled` + loading

## A6. 代码风格

| 项 | 标准 |
|----|------|
| 引号 | 单引号 |
| 分号 | 无 |
| 缩进 | 2 空格 |
| 尾随逗号 | 多行使用 |

导入顺序：`'use client'` → React/Next → 第三方 → `@/` → 相对 → `import type`

## A7. 质量门禁

```bash
yarn lint
yarn typecheck
yarn build
```

PR 最低要求：lint + typecheck 通过；无新增 `any`、`alert`、硬编码 hex。

# Part B 高级专题

复杂场景选用。不触发场景时不必过度设计。

## B1. RSC 缓存与失效

### 何时用

- 静态/半静态数据（个人资料、项目列表）可缓存
- mutation 后需刷新列表/详情

### 策略选型

| 手段 | 适用 |
|------|------|
| `fetch(..., { next: { revalidate: N } })` | 外部 HTTP，允许 N 秒 stale |
| `export const revalidate = N` | 整页 ISR |
| `revalidatePath('/knowledge')` | mutation 后刷新某路由 |
| `revalidateTag('kb-records')` | 细粒度，fetch 时 `tags: ['kb-records']` |
| `unstable_cache(fn, keys, { revalidate })` | 昂贵计算/聚合查询 |

### 原则

- Supabase 直查默认**不**跨请求缓存；需缓存时封装到 `lib/<domain>/queries.ts` 并显式标注 revalidate
- mutation 后**必须**调用 `revalidatePath` / `revalidateTag`，避免 stale UI
- 用户私有数据慎用长 revalidate；优先 `cache: 'no-store'` 或短 TTL

```tsx
// Server Action 示例
'use server'
import { revalidatePath } from 'next/cache'

export async function deleteRecord(id: string) {
  // ... delete
  revalidatePath('/knowledge')
}
```

## B2. Suspense、流式与 Error Boundary

### 分层

```
layout.tsx
  └── error.tsx          # 路由级 Error Boundary
  └── loading.tsx        # 路由级 Suspense fallback
  └── page.tsx
        └── <Suspense fallback={...}>
              └── AsyncChild   # 独立 await 的数据块
```

### 原则

- 慢查询拆成独立 async 子组件 + `<Suspense>`，避免整页阻塞 TTFB
- `error.tsx` 必须 `'use client'`，提供重试（`reset()`）
- 不要用 Suspense 包裹需 SEO 的首屏关键内容（应 SSR 直出）
- Client 子树局部错误：组件级 Error Boundary（`react-error-boundary` 或自写）

## B3. 并发与乐观更新

### 何时用

- 删除/点赞/切换状态等可预测 UI 的操作
- 用户期望即时反馈

### 模式

```tsx
'use client'
import { useTransition, useOptimistic } from 'react'

const [optimisticRecords, applyOptimistic] = useOptimistic(records, (state, deletedId: string) =>
  state.filter(r => r.id !== deletedId),
)
const [isPending, startTransition] = useTransition()

function handleDelete(id: string) {
  startTransition(async () => {
    applyOptimistic(id)
    const { error } = await deleteRecordAction(id)
    if (error) {
      // 回滚由 revalidate 或显式 refresh 恢复
      setError(error.message)
    }
  })
}
```

### 原则

- 乐观 UI 必须有失败回滚路径（revalidate 或 refetch）
- `isPending` 用于防重复提交，不用 `setTimeout` 假锁
- 复杂表单编辑不做乐观更新，用 loading + 明确成功/失败

## B4. 大列表与虚拟化

### 阈值

| 量级 | 策略 |
|------|------|
| < 50 | 普通 map 渲染 + 分页 |
| 50–500 | 分页 / 「加载更多」+ cursor |
| > 500 或单行很高 | 虚拟滚动（`@tanstack/react-virtual` 等） |

### 原则

- 虚拟列表项需固定或可测量高度，否则 scroll 跳动
- 服务端仍分页，虚拟化只解决 DOM 节点数，不替代 API 分页
- 图片列表配合 lazy load / `loading="lazy"`

## B5. Route Handler 安全模型

本仓库 RAG 等 API 可参考 `src/app/api/rag/chat/route.ts`。

### 必做清单

- [ ] 请求体大小上限（`content-length` 或读流截断）
- [ ] 字段长度 / 类型校验
- [ ] Rate limit（IP 或 token 桶）
- [ ] 不记录/不回显密钥、system prompt
- [ ] 统一错误 JSON：`{ error: string }` + 正确 HTTP status
- [ ] `service role` 仅服务端；敏感操作需鉴权（session / secret header）

```typescript
// ✅ 结构化错误
return NextResponse.json({ error: 'Rate limit exceeded' }, { status: 429 })

// ❌ 泄露内部信息
return NextResponse.json({ error: error.stack }, { status: 500 })
```

## B6. 性能与包体积

- 重组件（Matter.js、GSAP 场景）用 `next/dynamic` + `ssr: false`
- 图片：`next/image` 或明确 dimensions 防 CLS
- 避免 Client 边界过大导致 hydration 成本上升
- 发版前：`yarn build` 查看 First Load JS；单路由 chunk 异常膨胀时拆 dynamic import
- React Compiler 已开启，避免手动 `useMemo`/`useCallback` 除非 profiling 证明必要

## B7. 安全深化

- Markdown：`react-markdown` + 受限组件映射，禁止 `dangerouslySetInnerHTML`
- CSP：生产环境考虑限制 inline script（Next 默认有权衡，变更需评估）
- Signed URL：假设过期；长驻页 refresh 或 403 占位 + 重试
- 环境变量：`NEXT_PUBLIC_*` 仅公开值；密钥仅 Route Handler / scripts

# Part C 项目定制

仅 Tech-Centric 仓库。基线之上必须遵守。

## C1. SpiritGarden 样式体系

设计 token 与语义类定义于 [src/app/globals.css](../src/app/globals.css)。

### 优先级

1. **`sg-*` 语义类** — 表单、按钮、卡片、子页、模态框
2. **Tailwind utility** — 间距、flex/grid、响应式
3. **禁止** 组件内硬编码 hex（token 定义处除外）

```tsx
// ✅ GOOD
<button className="sg-btn sg-btn--primary">保存</button>
<div className="mt-4 text-[var(--sg-ink)]">...</div>

// ❌ BAD
<button style={{ background: '#154212' }}>保存</button>
```

### 新类命名

```
sg-<模块>-<元素>[--修饰符]
```

示例：`sg-kb-error--inline`、`sg-form-input`、`sg-btn--primary`

### 暗色模式

- `:root` / `.dark` CSS 变量切换
- 组件不写 `.dark:bg-[#xxx]`，用 `var(--color-*)` 或 `sg-*`

### Tailwind 4

- 配置在 `globals.css`：`@import "tailwindcss"` + `@theme inline`
- 不新增 `tailwind.config.js`

### 子页布局

```tsx
<div className="spirit-garden-content sg-subpage sg-subpage--archive">
```

### 错误展示

使用项目既有类：`sg-kb-error`、`sg-kb-error--inline`

## C2. Supabase 分端

| 环境 | 导入 |
|------|------|
| Server Component / Route Handler | `@/lib/supabase/server` |
| Client Component | `@/lib/supabase/client` |

```tsx
const { data, error } = await supabase.from('kb_records').select('*')
if (error) return <ErrorFallback message="加载失败，请刷新重试" />
const records = data ?? []
```

- RLS 为默认安全边界；`service role` 仅 Route Handler / scripts
- 上传前校验 `file.size`、MIME（建议上限 5MB，按模块可配置）
- 类型命名：**不用 `I` 前缀**（`KbRecord` 而非 `IRecord`）

## C3. 模块目录约定

| 模块 | 组件目录 | lib |
|------|----------|-----|
| 首页 | `components/home/` | — |
| 知识库 | `components/knowledge/` | `lib/knowledge/`（types、queries） |
| RAG 助手 | `components/rag/`（仅 UI） | `lib/rag/`（含 contactFlow） |
| 设计系统 | `components/spirit/` | — |

**划分原则**：按功能域分 `components/<domain>/`；非 UI 逻辑放 `lib/<domain>/`；跨域复用 UI 放 `components/spirit/`。

## C4. 已知偏离项（历史债）

**新代码禁止**；改旧文件时仅修正触达区域。

| 偏离 | 位置 | 目标 |
|------|------|------|
| 引号/分号混用 | 全仓库 | 单引号、无分号（A6） |
| Supabase tags 全量扫描 | `lib/knowledge/queries.ts` | 数据量大时改 RPC 聚合 |
